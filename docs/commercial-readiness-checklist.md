# 拳馆约课小程序商用上线清单

最后整理：2026-06-13

本文档用于判断当前拳馆约课小程序是否具备商用上线条件，并把剩余工作拆成可执行清单。它不是法律意见；涉及资质、隐私合规、合同、支付、退费、知识产权等事项，应由商家主体和专业人士最终确认。

## 结论

当前项目适合作为商用二开的主线，不建议直接替换成 GitHub 上的第三方预约小程序代码。

原因：

- 本项目已经围绕拳馆核心业务建模：多门店、会员、门店会员关系、课程、预约、课时余额、消课、微信账号绑定、后台角色权限。
- 外部仓库多数没有明确 License，不能默认用于商用二开。
- 部分有 License 的项目也可能存在额外 README 限制、AGPL 开源义务、旧技术栈、后台过重或行业不匹配问题。
- 当前项目的前端风格更接近拳馆，后端业务规则也比通用预约模板更接近真实运营。

商用前的主要缺口是：会员管理、课包/课时管理、课程状态体验、取消规则、通知规则、真实生产配置、隐私与上线材料。

## 优先级定义

- P0：商用上线前必须完成。不完成会影响安全、合规、数据正确性或基本运营。
- P1：外部试运营前强烈建议完成。不完成会增加前台人工成本或影响用户体验。
- P2：上线后增强。不是首发阻断项，但会影响长期运营效率。

## P0 上线阻断项

### 1. 商用主体与微信小程序生产环境

- [ ] 使用拳馆自己的小程序主体，不使用开发者个人 AppID 做正式运营。
- [ ] 完成微信认证、备案、类目选择、基本信息、头像、名称、简介配置。
- [ ] 确认服务类目适合拳馆约课业务，例如体育、体育培训、健身相关类目；最终以微信后台可选类目和审核要求为准。
- [ ] 配置生产 AppID、AppSecret、订阅消息模板 ID、API 域名等环境变量。
- [ ] AppSecret 只保存在后端环境变量中，不能出现在小程序前端代码、Git 仓库或截图中。
- [ ] 生产 API 使用 HTTPS 域名。
- [ ] 在微信公众平台配置 request 合法域名。
- [ ] 生产域名完成 ICP 备案或满足微信当前域名审核要求。
- [ ] 提交审核前完成微信后台隐私保护指引配置。

官方复核入口：

- [微信小程序备案操作指引](https://developers.weixin.qq.com/miniprogram/product/record_guidelines.html)
- [微信小程序网络/服务器域名文档](https://developers.weixin.qq.com/miniprogram/dev/framework/ability/network.html)
- [微信小程序用户隐私保护指引填写说明](https://developers.weixin.qq.com/miniprogram/dev/framework/user-privacy/)

### 2. 授权与知识产权

- [ ] 不直接复制无 License 的第三方 GitHub 项目代码用于商用。
- [ ] 不直接使用第三方项目中的图片、图标、海报、教练照片、门店照片、二维码、演示素材。
- [ ] 所有品牌元素、Logo、拳馆照片、教练照片、课程图片都有商用授权。
- [x] 使用的字体、图标库、UI 库、依赖包满足商用许可。
- [ ] 如参考第三方开源项目，只参考业务流程和交互思路，避免复制代码和素材。
- [x] 记录主要依赖和许可证，形成 `THIRD_PARTY_NOTICES` 或同类说明。

参考：

- [GitHub: Licensing a repository](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/licensing-a-repository)

当前证据：`THIRD_PARTY_NOTICES.md` 已记录当前 workspace 直接运行依赖和开发/构建依赖的包名、版本、用途和许可证，覆盖 `@nestjs/*`、`@prisma/client`、`antd`、`lucide-react`、`@tarojs/*`、`react` 等主要依赖。直接依赖许可证当前为 MIT、Apache-2.0 或 ISC；`pnpm ops:third-party-notices:test` 固定 notices 文档必须包含主要依赖、许可证和商用素材边界。该文件同时声明生产 Logo、拳馆照片、教练照片、课程图片、海报和二维码必须由商家或权利人提供商用授权。

### 3. 会员身份与课时管理

- [x] 关闭生产环境的宽松自动开户，或限制为邀请/白名单/后台确认后绑定。
- [x] 后台可以创建会员。
- [x] 后台可以绑定会员手机号或内部会员编号。
- [x] 后台可以给会员开课包、增加课时、扣减/调整课时。
- [x] 课时调整必须有操作人、原因、时间和变更前后数量。
- [x] 同一会员在不同门店的课时余额边界清楚。
- [x] 明确是否支持跨门店通用课包；如果首发不支持，前端和后台都要避免造成误解。
- [x] 会员换微信、绑定错微信账号时，有后台解绑和重绑流程。
- [x] 会员换手机号时，有后台编辑手机号流程。

当前证据：首发明确不支持跨门店通用课包，课时余额按 `userId + branchId` 隔离，会员端选择当前门店后只显示该门店课程、预约、剩余课时和消课记录；后台会员列表、课时调整和课时流水都带门店上下文。小程序“约课规则”已说明“课时按当前门店独立计算，暂不支持跨门店通用课包”，避免会员误以为课包可跨店使用。未来如要做跨门店通用课包，应作为 P2 单独建模和验收。

### 4. 用户端约课体验

- [x] 首页按日期分组展示课程，例如“今天、明天、本周”。
- [x] 课程卡片清楚显示：课程名、开始时间、时长、教练、门店、剩余名额、强度或适合人群。
- [x] 名额文案使用普通用户能理解的表达，例如“剩 3 位 / 共 8 位”。
- [x] 已预约的课程在列表中显示“已预约”，不能继续显示成普通“立即预约”。
- [x] 满员课程显示“已满”，按钮禁用。
- [x] 已取消或已开课课程不出现在可预约列表，或有明确状态。
- [x] 预约成功后给出明确反馈，并能直接去“我的预约”查看。
- [x] 取消预约前显示确认弹窗和取消规则。
- [x] 课程详情页包含训练内容、适合人群、装备要求、取消规则、门店地址、教练信息。
- [x] 空状态、加载中、网络错误、登录失败都有可理解文案。
- [x] 小程序 tab 命名清楚，建议为“约课 / 预约 / 账户”或“约课 / 我的预约 / 我的”。

### 5. 预约和取消业务规则

- [x] 后端继续强制检查用户身份，不能信任前端传入的 `userId`。
- [x] 后端继续检查门店访问权限。
- [x] 预约时检查：课程存在、属于当前门店、未取消、未开课、有剩余课时、未重复预约、未满员。
- [x] 预约容量检查保持事务或锁机制，避免并发超卖名额。
- [x] 明确取消截止时间，例如开课前 2 小时可取消，之后不可取消或需管理员处理。
- [x] 后端实现取消截止时间规则。
- [x] 取消后跳过或取消对应的待发送提醒任务。
- [x] 课程取消后，需要同步处理该课程下的有效预约状态和通知。
- [x] 明确爽约规则：是否扣课、谁来标记、是否可恢复。

当前证据：`CreateBookingDto` 不接受 `userId`，预约接口只使用 JWT 中的 `user.sub`；`BookingsService` 对会员门店、课程门店、课程状态、开课时间、课时余额、重复预约、容量和取消截止时间做服务端校验；容量检查在事务中通过 `SELECT ... FOR UPDATE` 锁定课程行。`ClassesService.listAvailable` 只返回 `SCHEDULED` 且 `startsAt > now` 的课程，E2E 已覆盖已取消和已开课课程不会进入用户端课程列表；小程序端也有 `isBookableClass` / `filterBookableClasses` 防御过滤。小程序课程列表按“今天 / 明天 / 本周 / 具体日期”分组展示；课程卡展示课程名、开始时间、教练、剩余名额、时长、门店、适合标签和课程描述，并可进入课程详情页。详情页展示训练内容、适合人群、装备要求、取消规则、门店地址、门店电话和教练信息，并支持直接预约；课程、预约、账户和详情页均使用统一加载、空状态、错误状态组件，加载中有“加载中，请稍候”文案，错误状态提供重新加载动作，登录/绑定失败会通过 `formatApiError` 展示可理解提示；小程序请求层统一设置 10 秒超时，并把 `timeout`、`request:fail` 和网络失败归一化为中文提示，`pnpm miniapp:network-errors:test` 固定该约束。tab 为“约课 / 预约 / 账户”，避免“我的”和“账户”混淆；预约成功后显示“预约成功”确认弹窗，可选择“查看预约”直接切到“我的预约”，也可“继续约课”；取消预约弹窗会在确认前显示课程、时间、开课前 2 小时取消规则、截止后联系拳馆处理，以及取消成功后释放名额并停止待发送提醒。爽约不自动扣课，截止后或爽约由工作人员按馆规处理；如需要扣课，由管理员/店长在后台对该预约执行消课；如误扣或需要恢复，通过课时调整流水纠正并保留操作记录。E2E 已覆盖跨门店预约、门店不匹配、重复预约、满员、并发最后一名额、取消截止和取消他人预约。

### 6. 消课与财务边界

- [x] 管理员只能消自己有权限门店的预约。
- [x] 取消预约不能消课。
- [x] 同一预约只能消课一次。
- [x] 消课必须事务处理：创建消课记录、更新预约到课状态、扣减课时余额。
- [x] 课时不足时不能消课，并给管理员清楚提示。
- [x] 消课记录不可硬删除；如需纠错，应走冲正或调整记录。
- [x] 后台显示会员当前剩余课时，避免前台消课时盲操作。

当前证据：消课接口只允许 `ADMIN`，并通过 `ensureAdminBranchRole` 限定 `OWNER/MANAGER` 所属门店；`LessonDeductionsService.deductLesson` 在同一事务里检查预约状态、重复消课、课时余额，创建 `LessonDeduction`，扣减 `LessonBalance`，并把预约到课状态改为 `ATTENDED`。`LessonDeduction.bookingId` 有唯一约束，生产代码没有消课记录删除接口；纠错可用课时调整流水。

### 7. 后台运营能力

- [x] 后台预约消课 tab 提供今日课程、预约人数和待消课人数的运营视图。
- [x] 后台可以按日期、门店、课程、会员姓名、手机号、预约状态筛选。
- [x] 后台可以创建、编辑、取消课程。
- [x] 编辑课程容量时，不能低于已预约人数。
- [x] 取消课程前显示影响人数。
- [x] 取消课程后能触发会员通知或生成待通知任务。
- [x] 后台可以管理教练基础信息，至少包括姓名、可见昵称、所属门店、状态。
- [x] 管理员、店长、教练的权限边界清楚。
- [x] 店长不能查看或操作无权限门店。

当前证据：后台预约查询支持 `branchId`、营业日 `date`、状态 `status` 和关键词 `q`，关键词覆盖会员姓名、手机号和课程名；预约消课 tab 已提供“今日课程”快捷筛选和按课程分组的预约名单卡，展示预约人数和待消课人数，可展开逐个消课或取消预约。课程接口已覆盖创建、编辑、取消，容量低于有效预约数时拒绝；取消课程确认框会展示当前已预约会员数，并提示会生成课程取消通知任务。后端取消课程会在同一事务中取消有效预约、跳过原开课提醒、为每个被影响会员生成 `CLASS_CANCELED` 通知任务，并在 `CLASS_CANCEL` 审计日志里记录 `affectedBookingCount` 和 `notificationJobCount`。后台新增 `/admin/coaches` 教练管理，支持创建、列表和启停教练档案，字段包含姓名、可见昵称、所属门店、账号、手机号和状态；排课可选择 `coachId` 绑定教练档案，并保留展示教练名快照。管理员/店长可在所属门店管理教练、课程和消课；教练账号只能查看自己负责的课程和预约，不能创建课程或执行消课。店长角色的预约、消课、审计日志和通知任务列表均按门店范围过滤或拒绝越权。`pnpm admin:class-cancel:test`、`pnpm admin:coaches:test` 和 API E2E 已覆盖该链路。

### 8. 隐私、协议与用户告知

- [ ] 微信后台隐私保护指引中声明实际收集的数据类型。

当前证据：`docs/data-retention-policy.md` 已列出会员身份资料、微信绑定、门店权限、课程预约、课时消课、通知任务、审计日志和管理员账号的数据用途，并明确首发不应收集与约课无关的数据。小程序账户页已提供“隐私政策”和“约课规则”入口，隐私弹窗说明必要数据用途和不收集身份证、精确定位、通讯录、相册、麦克风、摄像头等无关信息，约课规则弹窗说明取消截止、消课和课程取消处理。`pnpm security:check` 已接入 `scripts/check-sensitive-logging.mjs`，会拒绝日志调用中直接输出 `openid`、手机号、JWT/token、AppSecret、密码等敏感字段；通知失败日志在落库前会脱敏 `openid`、`access_token`、`secret`、手机号等第三方错误详情，并有 E2E 覆盖。微信后台隐私保护指引配置仍需上线前完成。

当前证据：后端已有 `GET /admin/notifications` 和 `POST /admin/notifications/:id/retry`，通知任务按门店权限过滤，返回最新发送日志 `latestLog`，支持按 `FAILED`/`SKIPPED` 等状态筛选，并在重试后写入发送结果日志和 `NOTIFICATION_RETRY` 审计日志。小程序预约前会按已配置模板请求订阅授权：`TARO_APP_WECHAT_BOOKING_CREATED_TEMPLATE_ID` 对应预约确认，`TARO_APP_WECHAT_SUBSCRIBE_TEMPLATE_ID` 对应开课提醒；后端在用户授权预约确认时创建 `BOOKING_CREATED` 通知任务，模板 ID 由 `WECHAT_BOOKING_CREATED_TEMPLATE_ID` 配置。后台“通知任务”页展示失败原因、失败状态筛选、通知类型和重试按钮，可区分“预约确认”“开课提醒”“课程取消通知”和“课程改期通知”。取消课程会生成 `CLASS_CANCELED` 通知任务，待 worker 发送或在模板缺失时标记跳过；课程改期会为每个有效预约生成 `CLASS_RESCHEDULED` 通知任务，并把原有待发送开课提醒按旧提前量移动到新上课时间前。`pnpm notifications:booking-confirmation:test` 固定预约确认通知入口，API E2E 覆盖预约后生成可配置确认通知任务；`pnpm admin:notifications:test` 固定后台入口，API E2E 覆盖失败日志可见、店长门店隔离和失败任务重试；`pnpm admin:class-cancel:test` 和 API E2E 覆盖课程取消通知任务生成；`pnpm admin:class-reschedule:test` 和 API E2E 覆盖课程改期通知任务生成与提醒时间移动。

## 上线 Go / No-Go 标准

满足以下条件才建议商用发布：

- [ ] 会员可以用真实微信登录并绑定到正确会员。
- [ ] 会员只能看到自己所属门店、自己的预约、自己的课时。
- [ ] 会员可以完成约课、取消、查看预约和消课记录。
- [ ] 管理员可以排课、查预约、消课、查看消课记录。
- [ ] 店长权限不能越权到其他门店。
- [ ] 并发预约不会超卖名额。
- [ ] 重复预约和重复消课会被拒绝。
- [x] 生产环境没有默认管理员密码。
- [x] 生产环境没有测试 seed 数据。
- [ ] 生产 AppID、域名、隐私指引、备案、类目配置完成。
- [ ] 数据库有备份。
- [ ] 至少完成一轮真实手机端端到端验收。

如果以上任一项不满足，建议只做内部测试，不建议对真实会员开放。
